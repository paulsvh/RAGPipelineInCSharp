# On-Call Runbook

**Document Owner:** SRE Team
**Last Updated:** March 2024
**Classification:** Internal (see [Data Retention Policy](../policies/data-retention-policy.md))

## Overview

This runbook provides step-by-step remediation procedures for common alerts generated by the [Pulse Monitoring Suite](../products/pulse-monitoring-suite.md). It is intended for on-call engineers responding to production incidents as defined in the [Incident Response Plan](../policies/incident-response-plan.md).

Before beginning any remediation, ensure you have:

1. Acknowledged the alert in PagerDuty
2. Opened an incident Slack channel (for SEV1/SEV2)
3. Verified VPN connectivity (required per the [Acceptable Use Policy](../policies/acceptable-use-policy.md))
4. Access to the production Kubernetes cluster (see [ADR-003](../engineering/architecture-decision-records.md))

## Alert: High CPU on API Servers

**Pulse Alert Name:** `api-server-cpu-high`
**Threshold:** CPU utilization > 85% for 5 consecutive minutes
**Severity:** Typically SEV2 or SEV3 depending on customer impact
**Affected Services:** Aurora Analytics API, Nebula Cloud Storage API, Pulse API

### Diagnosis

**Step 1: Identify the affected pods and nodes.**

```bash
kubectl top pods -n aurora --sort-by=cpu | head -20
kubectl top nodes | sort -k3 -rn | head -10
```

**Step 2: Check if a recent deployment triggered the spike.**

```bash
kubectl rollout history deployment/aurora-api -n aurora
helm history aurora-api -n aurora --max 5
```

If a deployment occurred within the last 30 minutes, it is likely the cause. Proceed to rollback (Step 5).

**Step 3: Check for runaway queries (Aurora-specific).**

Connect to the PostgreSQL primary and look for long-running queries. Connection details are in the `aurora-db-credentials` Kubernetes secret.

```bash
kubectl exec -it deployment/psql-client -n aurora -- psql -h $DB_HOST -U aurora_admin -d aurora_production -c "
SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
FROM pg_stat_activity
WHERE state != 'idle'
AND now() - pg_stat_activity.query_start > interval '30 seconds'
ORDER BY duration DESC;
"
```

Cancel any runaway queries:

```bash
kubectl exec -it deployment/psql-client -n aurora -- psql -h $DB_HOST -U aurora_admin -d aurora_production -c "
SELECT pg_cancel_backend(<PID>);
"
```

**Step 4: Scale horizontally.**

If the load is legitimate (not caused by a bad query or deployment), scale the deployment:

```bash
kubectl scale deployment/aurora-api -n aurora --replicas=8
```

Verify that the Horizontal Pod Autoscaler (HPA) is functioning:

```bash
kubectl get hpa -n aurora
```

If the HPA is not scaling, check that the metrics-server is running:

```bash
kubectl get pods -n kube-system | grep metrics-server
```

**Step 5: Rollback (if deployment-related).**

```bash
kubectl rollout undo deployment/aurora-api -n aurora
```

Verify the rollback:

```bash
kubectl rollout status deployment/aurora-api -n aurora
```

## Alert: Database Connection Pool Exhaustion

**Pulse Alert Name:** `db-connection-pool-exhausted`
**Threshold:** Available connections < 5 for 2 consecutive minutes
**Severity:** SEV1 if all connections exhausted; SEV2 if approaching limit
**Affected Services:** All services using PostgreSQL (see [ADR-001](../engineering/architecture-decision-records.md))

### Diagnosis

**Step 1: Check current connection count in PgBouncer.**

```bash
kubectl exec -it deployment/pgbouncer -n aurora -- psql -p 6432 -U pgbouncer pgbouncer -c "SHOW POOLS;"
```

**Step 2: Identify which service is consuming the most connections.**

```bash
kubectl exec -it deployment/psql-client -n aurora -- psql -h $DB_HOST -U aurora_admin -d aurora_production -c "
SELECT application_name, count(*)
FROM pg_stat_activity
GROUP BY application_name
ORDER BY count DESC;
"
```

**Step 3: Check for connection leaks.**

Look for connections in an `idle in transaction` state that have been open for an unusually long time:

```bash
kubectl exec -it deployment/psql-client -n aurora -- psql -h $DB_HOST -U aurora_admin -d aurora_production -c "
SELECT pid, application_name, state, now() - state_change AS idle_duration
FROM pg_stat_activity
WHERE state = 'idle in transaction'
AND now() - state_change > interval '5 minutes'
ORDER BY idle_duration DESC;
"
```

Terminate leaked connections:

```bash
kubectl exec -it deployment/psql-client -n aurora -- psql -h $DB_HOST -U aurora_admin -d aurora_production -c "
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle in transaction'
AND now() - state_change > interval '10 minutes';
"
```

**Step 4: Increase pool size (temporary).**

Update the PgBouncer configuration to increase the pool size:

```bash
kubectl edit configmap pgbouncer-config -n aurora
# Increase max_client_conn and default_pool_size
kubectl rollout restart deployment/pgbouncer -n aurora
```

**Step 5: Review slow queries.**

Slow queries hold connections longer, accelerating pool exhaustion. Check the slow query log:

```bash
kubectl exec -it deployment/psql-client -n aurora -- psql -h $DB_HOST -U aurora_admin -d aurora_production -c "
SELECT query, calls, mean_exec_time, total_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 20;
"
```

## Alert: Certificate Expiration Warning

**Pulse Alert Name:** `tls-cert-expiring`
**Threshold:** Certificate expires within 14 days
**Severity:** SEV3 (14 days), escalates to SEV2 (7 days), SEV1 (< 24 hours)
**Affected Services:** All public-facing endpoints

### Diagnosis

**Step 1: Identify which certificate is expiring.**

```bash
kubectl get certificates -A | grep -v True
```

Check certificate details:

```bash
kubectl describe certificate <cert-name> -n <namespace>
```

**Step 2: Check cert-manager status.**

Most certificates are managed by cert-manager and should auto-renew. If renewal failed, check the cert-manager logs:

```bash
kubectl logs deployment/cert-manager -n cert-manager --tail=100 | grep -i error
```

Common causes of renewal failure:
- DNS challenge authorization failure (check Route53 permissions)
- Rate limiting by Let's Encrypt (check order status)
- cert-manager pod crash loop

**Step 3: Force renewal via cert-manager.**

```bash
kubectl delete certificate <cert-name> -n <namespace>
# cert-manager will automatically recreate and renew
kubectl get certificaterequests -n <namespace> -w
```

**Step 4: Manual renewal with certbot (fallback).**

If cert-manager is not functioning, renew manually:

```bash
certbot certonly --dns-route53 -d "*.vortextech.io" -d "vortextech.io" --non-interactive --agree-tos
```

**Step 5: Update Kubernetes secrets.**

If the certificate was renewed manually, update the TLS secret:

```bash
kubectl create secret tls vortextech-tls \
  --cert=/etc/letsencrypt/live/vortextech.io/fullchain.pem \
  --key=/etc/letsencrypt/live/vortextech.io/privkey.pem \
  -n ingress --dry-run=client -o yaml | kubectl apply -f -
```

**Step 6: Verify rotation.**

```bash
echo | openssl s_client -connect aurora.vortextech.io:443 -servername aurora.vortextech.io 2>/dev/null | openssl x509 -noout -dates
```

## Alert: Storage Quota Exceeded on Nebula

**Pulse Alert Name:** `nebula-storage-quota-exceeded`
**Threshold:** Bucket usage > 90% of allocated quota
**Severity:** SEV3 (90%), SEV2 (95%), SEV1 (100% — writes rejected)
**Affected Services:** [Nebula Cloud Storage](../products/nebula-cloud-storage.md), any service writing to Nebula

### Diagnosis

**Step 1: Identify which bucket is exceeding quota.**

```bash
curl -s -H "Authorization: Bearer $NEBULA_ADMIN_TOKEN" \
  "https://api.vortextech.io/v1/nebula/admin/buckets?sort=usage_percent&order=desc&limit=10" | jq '.buckets[] | {name, usage_bytes, quota_bytes, usage_percent}'
```

**Step 2: Check for log accumulation.**

Application logs are a common cause of unexpected storage growth. Check the largest prefixes in the offending bucket:

```bash
aws s3api list-objects-v2 \
  --bucket <bucket-name> \
  --endpoint-url https://nebula.vortextech.io \
  --prefix "logs/" \
  --query "sort_by(Contents, &Size)[-10:].[Key, Size]" \
  --output table
```

**Step 3: Archive old data.**

Move data older than 90 days to the Archive tier using Nebula's lifecycle API:

```bash
curl -X PUT -H "Authorization: Bearer $NEBULA_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  "https://api.vortextech.io/v1/nebula/buckets/<bucket-name>/lifecycle" \
  -d '{
    "rules": [{
      "id": "archive-old-logs",
      "prefix": "logs/",
      "transition": {"days": 90, "storage_class": "ARCHIVE"},
      "expiration": {"days": 365}
    }]
  }'
```

**Step 4: Increase quota via admin API.**

If the data is legitimate and retention is required (check the [Data Retention Policy](../policies/data-retention-policy.md)), increase the bucket quota:

```bash
curl -X PATCH -H "Authorization: Bearer $NEBULA_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  "https://api.vortextech.io/v1/nebula/admin/buckets/<bucket-name>/quota" \
  -d '{"quota_bytes": 5368709120}'
```

**Step 5: Verify.**

Confirm the quota has been updated and the alert has cleared in Pulse:

```bash
curl -s -H "Authorization: Bearer $NEBULA_ADMIN_TOKEN" \
  "https://api.vortextech.io/v1/nebula/admin/buckets/<bucket-name>" | jq '{name, usage_bytes, quota_bytes, usage_percent}'
```

## General Escalation Notes

- If any alert cannot be resolved within the expected response time for its severity level, escalate per the [Incident Response Plan](../policies/incident-response-plan.md).
- All remediation actions taken during an incident should be documented in the incident Slack channel for inclusion in the post-mortem.
- For issues related to inter-service communication failures, check gRPC connectivity — see [ADR-002](../engineering/architecture-decision-records.md) for architecture context.
- Rate limiting issues on public APIs follow the thresholds documented in the [API Style Guide](../engineering/api-style-guide.md).
